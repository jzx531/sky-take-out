这个错误信息 `Could not create server TCP listening socket 127.0.0.1:6379: bind: 操作成功完成` 看起来非常矛盾（“无法创建”却提示“操作成功”），但这在 Windows 版的 Redis 中是一个已知的常见现象。

**核心原因：**
这通常意味着 **端口 6379 已经被占用了**。
最常见的情况是：你之前已经启动过一个 Redis 实例（可能在后台运行，或者之前的窗口没关干净），它已经占用了 6379 端口。当你尝试再次启动时，新的实例无法绑定该端口，从而报错。那个“操作成功完成”通常是 Windows Socket API 返回的一个误导性状态码或日志打印错误，实际结果是绑定失败。

既然端口被占用，说明 **Redis 很可能已经在运行了**。你不需要“重连”或重新启动它，而是应该直接连接正在运行的那个实例。

以下是解决步骤：

### 方法一：直接使用客户端连接（推荐）

既然服务端可能已经在运行，你只需要打开一个新的命令行窗口（不要关闭当前报错的这个，先留着观察），然后运行客户端：

1.  保持当前的报错窗口不动。
2.  打开一个新的 `cmd` 或 `PowerShell` 窗口。
3.  进入 Redis 目录：
    ```cmd
    F:
    cd F:\Redis-x64-5.0.14.1
    ```
4.  尝试连接：
    ```cmd
    redis-cli.exe
    ```
5.  如果连接成功，你会看到提示符变为 `127.0.0.1:6379>`。此时输入 `ping`，如果返回 `PONG`，说明 Redis 服务正常，你可以直接使用了。

---

### 方法二：查找并关闭占用的进程

如果你想重新启动一个全新的 Redis 实例（例如为了加载新的配置文件），你需要先杀死占用端口的旧进程。

#### 步骤 1：找出占用 6379 端口的进程 ID (PID)
在命令行中运行以下命令：
```cmd
netstat -ano | findstr :6379
```
你会看到类似这样的输出：
```text
TCP    127.0.0.1:6379         0.0.0.0:0              LISTENING       12345
```
最后的数字（例如 `12345`）就是 **PID**。

#### 步骤 2：强制结束该进程
使用 `taskkill` 命令结束它（将 `12345` 替换为你实际查到的 PID）：
```cmd
taskkill /F /PID 12345
```
*如果提示成功终止，就可以回到原来的窗口重新运行 `redis-server.exe` 了。*

#### 备选方案：通过任务管理器
如果不喜欢用命令，也可以：
1.  按 `Ctrl + Shift + Esc` 打开任务管理器。
2.  点击“详细信息”选项卡。
3.  找到 `redis-server.exe`。
4.  右键点击 -> “结束任务”。

---

### 方法三：修改配置文件使用其他端口

如果你希望同时运行多个 Redis 实例，或者不想杀掉旧的进程，可以修改配置文件使用其他端口。

1.  用记事本打开 `F:\Redis-x64-5.0.14.1\redis.windows.conf`。
2.  搜索 `port 6379`。
3.  将其修改为其他端口，例如 `port 6380`。
4.  保存文件。
5.  重新启动服务：
    ```cmd
    redis-server.exe redis.windows.conf
    ```
6.  连接时指定端口：
    ```cmd
    redis-cli.exe -p 6380
    ```

### 总结
你遇到的错误大概率是因为 **Redis 已经在后台运行了**。
*   **最快解决方案**：直接新开一个窗口运行 `redis-cli.exe` 进行连接。
*   **如果想重启**：先用 `netstat` 找到 PID 并 `taskkill` 杀掉旧进程，再重新启动。
